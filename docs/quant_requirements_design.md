# 股票量化平台需求设计文档

## 1. 文档说明

### 1.1 文档目标

本文档用于从产品和业务需求角度完整定义一套股票量化平台，覆盖从数据接入、研究分析、策略开发、组合管理、回测验证、模拟交易、实盘执行到运维治理的全链路能力。

本文档不依赖当前项目实现细节，目标是给出一套更完整、更平台化、更接近生产级量化系统的需求设计。

### 1.2 文档定位

本文档回答以下问题：

1. 这个量化平台要服务哪些角色
2. 这个量化平台要解决哪些核心业务问题
3. 平台需要提供哪些完整功能
4. 每个功能模块的目标、输入、输出和验收标准是什么
5. 平台应该如何分阶段建设

### 1.3 关联文档

- `docs/quant_framework_system_design.md`
- `docs/quant_development_plan.md`

## 2. 产品愿景与建设目标

### 2.1 产品愿景

建设一套面向股票市场的机构级量化研究与交易平台，使研究、验证、交易、风控、运维形成统一闭环，支持从“策略想法”到“生产交易”的完整生命周期管理。

### 2.2 建设目标

1. 建立统一的数据和研究底座，避免研究结论无法复现
2. 建立标准化策略研发流程，降低策略交付成本
3. 建立统一组合和风控体系，减少策略上线风险
4. 建立完整订单和账户管理体系，打通交易闭环
5. 建立可追溯、可审计、可观测的运维与治理体系
6. 支持单团队研发，也支持未来多团队协同和多账户运营

### 2.3 平台定位

平台首期聚焦：

- A 股和美股
- 日频和准日频
- 中低频股票策略
- long-only、指数增强、多因子选股、行业轮动、质量动量、低波防御

平台中长期扩展：

- 多资产
- 市场中性
- 事件驱动
- 机器学习排序
- 统计套利与对冲框架

## 3. 用户角色与核心诉求

### 3.1 量化研究员

核心诉求：

- 快速搭建研究数据集
- 设计因子并验证有效性
- 快速迭代策略参数
- 获取样本内外和滚动验证结果
- 能解释策略收益来源和失效原因

### 3.2 投资经理 / 策略负责人

核心诉求：

- 管理策略版本和预设
- 审核策略稳定性和风险暴露
- 选择上线账户和资金规模
- 监控策略表现与偏离

### 3.3 交易员 / 交易运营

核心诉求：

- 获取清晰的调仓建议和订单列表
- 审批、提交、撤单、追单
- 实时查看订单状态和成交情况
- 处理异常订单和账户异常

### 3.4 风控负责人

核心诉求：

- 配置研究风控、组合风控和交易风控规则
- 查看风险预警、违规明细和风险趋势
- 审批例外豁免
- 对策略上线和实盘运行进行风险把关

### 3.5 数据工程师 / 平台工程师

核心诉求：

- 统一管理数据源和同步任务
- 保证数据质量和时效性
- 管理任务编排、日志、告警和资源消耗
- 快速定位数据或系统异常

### 3.6 管理员 / 运维

核心诉求：

- 管理用户、权限、环境和配置
- 统一维护任务、服务、调度和审计
- 保证系统安全、稳定和可恢复

## 4. 业务范围

### 4.1 纳入范围

- 市场数据、财务数据、基准数据、行业数据、公司行为数据管理
- 股票池筛选、因子开发、信号生成、组合构建、回测与验证
- 模拟盘和实盘交易管理
- 风险规则、审批流、审计与告警
- 报表中心、可视化和 API 服务

### 4.2 暂不纳入首期范围

- 分钟级高频交易
- 超低延迟撮合
- 期权定价和复杂衍生品交易
- 跨境清结算系统
- 高频 Tick 级深度簿研究

---

## 5. 业务流程总览

### 5.1 研究到上线主流程

1. 数据同步与质量校验
2. 研究员构建股票池和特征集
3. 因子研究与诊断
4. 形成策略版本和参数集
5. 回测、walk-forward、稳定性评估
6. 进入模拟盘观察
7. 风控审批
8. 发布到生产账户
9. 日常运行监控和绩效归因

### 5.2 每日生产运行流程

1. 开盘前完成数据更新
2. 执行策略生成目标组合
3. 执行组合风控和交易风控
4. 形成订单建议
5. 人工审批或自动执行
6. 券商回报同步
7. 更新账户和持仓
8. 生成日终归因和运维报告

### 5.3 异常处理流程

1. 发现数据异常、订单异常或风险异常
2. 系统自动告警并冻结相关任务
3. 运维或风控确认异常级别
4. 执行重试、降级、人工处理或回滚
5. 完成异常复盘和审计归档

## 7. 详细功能需求

## 7.1 数据接入与数据治理

### 7.1.1 目标

建立统一、稳定、可追溯的数据底座，为研究和交易提供可信数据。

### 7.1.2 功能要求

- 支持接入以下数据类型：
  - 日线行情
  - 复权行情
  - 基准指数
  - 财务报表
  - 估值指标
  - 行业分类
  - 公司行为
  - 交易日历
  - 券商账户与订单回报
- 支持多个外部数据源
- 支持源优先级、回退和熔断
- 支持全量同步和增量同步
- 支持断点续传
- 支持按市场、日期、标的批量拉取
- 支持数据源健康检查和时效监控
- 支持数据质量校验：
  - 缺失率
  - 重复率
  - 价格异常
  - 交易量异常
  - 财务字段口径异常
- 支持数据版本快照
- 支持字段 lineage 追踪

### 7.1.3 验收标准

- 同一交易日的数据可重复获取且结果一致
- 数据质量报告可定位到具体日期、数据源和字段
- 增量同步失败时可从断点恢复

## 7.2 证券主数据与市场规则

### 7.2.1 目标

统一管理证券、市场和交易规则，使策略与交易逻辑不直接依赖硬编码规则。

### 7.2.2 功能要求

- 管理证券主数据：
  - 股票代码
  - 市场
  - 交易所
  - 证券类型
  - 上市日期
  - 行业分类
  - 是否 ST
  - 是否可融资融券
- 管理市场规则：
  - 最小交易单位
  - 涨跌停规则
  - T+1 / T+0
  - 税费规则
  - 交易时段
  - 休市安排
- 支持规则版本化
- 支持按市场、资产类别和账户维度配置

### 7.2.3 验收标准

- 不同市场规则由配置驱动，不依赖策略硬编码
- 市场规则变更后可追溯历史版本

## 7.3 研究工作台

### 7.3.1 目标

提供统一研究入口，让研究员可以基于标准数据集和流程完成探索、验证和结论沉淀。

### 7.3.2 功能要求

- 支持创建研究项目
- 支持定义研究范围：
  - 市场
  - 日期区间
  - 股票池
  - 基准
- 支持特征计算和缓存
- 支持中间结果持久化
- 支持研究笔记和结论沉淀
- 支持研究结果共享和对比

### 7.3.3 验收标准

- 同一研究项目可重复运行
- 研究输入、参数和结果可完整追踪

## 7.4 因子平台

### 7.4.1 目标

统一管理因子定义、计算、诊断、版本和淘汰流程。

### 7.4.2 功能要求

- 因子注册与分类
- 因子公式管理
- 因子依赖字段管理
- 因子版本管理
- 因子计算任务
- 因子诊断能力：
  - IC / RankIC
  - 分层收益
  - 换手率
  - 覆盖率
  - 稳定性
  - 衰减分析
- 因子组合实验
- 因子评分卡
- 因子下线机制

### 7.4.3 验收标准

- 每个因子都可查询版本、依赖、诊断结果和适用市场
- 因子修改后可清晰区分新旧口径

## 7.5 策略工厂

### 7.5.1 目标

支持从策略定义、参数配置、版本生成到上线审批的完整策略生命周期。

### 7.5.2 功能要求

- 策略模板管理
- 策略分类管理：
  - 多因子
  - 指数增强
  - 行业轮动
  - 价值
  - 低波防御
  - 机器学习排序
- 策略输入参数管理
- 策略版本冻结
- 策略实验记录
- 策略表现评分卡
- 策略状态管理：
  - 草稿
  - 研究中
  - 待评审
  - 模拟中
  - 生产中
  - 已停用

### 7.5.3 验收标准

- 任意策略版本都可定位到对应数据版本、因子版本和参数版本
- 策略状态变更具备审批记录

## 7.6 组合管理与优化

### 7.6.1 目标

把策略信号转化为可执行的目标组合，并控制组合质量与风险。

### 7.6.2 功能要求

- 支持多种股票池来源：
  - 全市场
  - 指数成分股
  - 自定义股票池
  - 黑白名单过滤
- 支持多种打分机制：
  - 加权评分
  - 排名评分
  - 分位数评分
  - 模型预测分数
- 支持多种组合构建方式：
  - 等权
  - 分数加权
  - 风险预算
  - 基准增强
  - 优化器驱动
- 支持组合约束：
  - 单票权重上限
  - 行业权重上限
  - 风格暴露约束
  - Beta 约束
  - 流动性约束
  - 最大换手率
  - 现金缓冲

### 7.6.3 验收标准

- 每次组合生成都有完整目标持仓和诊断输出
- 组合结果能够解释约束如何生效

## 7.7 回测与验证中心

### 7.7.1 目标

为策略提供足够严谨的历史验证体系，避免仅靠单区间回测做决策。

### 7.7.2 功能要求

- 单期回测
- 滚动回测
- train / validate / test 切分
- walk-forward 验证
- 参数稳定性分析
- 因子消融分析
- 交易成本回放
- 资金规模敏感性分析
- 基准对比
- 情景回放：
  - 牛市
  - 熊市
  - 震荡市
  - 极端波动期
- 多策略横向比较

### 7.7.3 验收标准

- 任意策略上线前必须至少具备样本外验证和稳定性分析
- 回测结果需可导出结构化报告

## 7.8 订单管理与执行中心

### 7.8.1 目标

把组合计划转化为可控订单，并打通提交、成交、回写和异常处理闭环。

### 7.8.2 功能要求

- 调仓差分计算
- 订单建议生成
- 订单审批
- 订单拆分与合并
- 订单生命周期管理：
  - 创建
  - 审批
  - 提交
  - 部分成交
  - 全部成交
  - 取消
  - 拒绝
  - 过期
- 支持模拟撮合
- 支持券商订单路由
- 支持成交回报同步
- 支持撤单、补单和追单
- 支持订单异常分类和处理

### 7.8.3 验收标准

- 所有订单都能查到完整状态流转
- 订单与调仓计划、策略版本、账户能够建立完整映射

## 7.9 账户与资产管理

### 7.9.1 目标

支持多账户、组合和资金管理，形成统一账户视图。

### 7.9.2 功能要求

- 多账户管理
- 多策略绑定
- 账户净值与收益追踪
- 持仓快照管理
- 资金分配管理
- 账户约束配置
- 模拟盘与实盘账户区分
- 账户级绩效归因

### 7.9.3 验收标准

- 多账户运行互相隔离
- 单账户所有持仓、订单、收益可追踪

## 7.10 风险管理中心

### 7.10.1 目标

建立覆盖研究、组合、交易和运行过程的全链路风险控制体系。

### 7.10.2 功能要求

- 研究风控：
  - 防止样本泄漏
  - 防止过拟合
  - 监控参数漂移
- 组合风控：
  - 集中度
  - 行业暴露
  - 风格暴露
  - Beta 风险
  - 容量风险
- 交易风控：
  - 可交易校验
  - 涨跌停
  - T+1
  - ST
  - 黑名单
  - 单笔订单上限
  - 参与率上限
- 运行风控：
  - 重复任务拦截
  - 重复发单拦截
  - 数据异常冻结
  - 券商接口异常降级
- 风险事件记录
- 风险白名单和豁免审批

### 7.10.3 验收标准

- 风险规则具备配置能力和审计记录
- 风险拦截必须能返回清晰原因

## 7.11 审批与发布治理

### 7.11.1 目标

确保策略和配置进入生产前具备可控审批链路。

### 7.11.2 功能要求

- 策略上线审批
- 参数变更审批
- 风险豁免审批
- 生产账户绑定审批
- 配置发布审批
- 回滚审批和记录

### 7.11.3 验收标准

- 所有关键生产变更具备审批记录和操作人

## 7.12 报表与分析中心

### 7.12.1 目标

为研究、交易、风控和管理层提供统一分析与汇报能力。

### 7.12.2 功能要求

- 策略收益报表
- 回测报告
- 因子诊断报告
- 风险报表
- 订单执行报表
- 账户净值报表
- 收益归因报表
- 成本拖累分析
- 异常事件复盘报告
- 导出 PDF / Markdown / JSON / CSV

### 7.12.3 验收标准

- 报表可按策略、账户、市场、日期多维筛选
- 报表支持结构化导出

## 7.13 运维与监控中心

### 7.13.1 目标

让系统具备可观测、可维护、可定位问题的生产能力。

### 7.13.2 功能要求

- 服务健康检查
- 任务调度管理
- 任务历史和运行日志
- 数据同步监控
- 风险事件监控
- 订单异常监控
- 账户异常监控
- 告警通知：
  - 邮件
  - IM
  - Webhook
- 运维面板
- 灰度开关
- 手动重跑和任务解锁

### 7.13.3 验收标准

- 关键任务失败可自动告警
- 关键状态可在统一运维面板查看

## 7.14 权限、组织与开放平台

### 7.14.1 目标

支持多角色协作和外部系统集成。

### 7.14.2 功能要求

- 用户与角色管理
- 数据权限管理
- 策略权限管理
- 账户交易权限
- 审批权限
- API Key 管理
- 开放 API
- Webhook 订阅
- 审计日志

### 7.14.3 验收标准

- 不同角色只能访问授权资源
- 所有关键操作具备审计记录

## 8. 非功能性需求

## 8.1 性能要求

- 全市场日频选股任务应在可接受时间内完成
- 常规日频回测应支持并行执行
- 订单与风险校验链路应低延迟返回
- 报表生成不应阻塞核心交易流程

## 8.2 可用性要求

- 核心服务应具备高可用部署能力
- 关键任务应具备自动重试与熔断机制
- 系统应支持故障恢复和人工接管

## 8.3 可扩展性要求

- 新市场接入不应重写全部业务逻辑
- 新因子、新策略、新券商应以插件化或注册式方式扩展
- 支持未来多资产扩展

## 8.4 可追溯性要求

- 所有研究结果、策略版本、参数版本、订单和账户变更都应可追溯
- 任意生产结果都应定位回对应输入快照

## 8.5 安全性要求

- 凭证和密钥必须加密存储
- 关键生产操作必须具备权限和审批控制
- 所有外部接口调用必须留痕

## 8.6 合规性要求

- 留存完整操作日志
- 留存订单和审批审计
- 留存风险拦截与例外审批记录

## 11. 阶段性验收标准

### 11.1 阶段一：研究可用

- 研究员可独立完成从数据到回测的闭环
- 策略结果可导出标准报告
- 因子和策略具备版本管理

### 11.2 阶段二：交易可用

- 模拟盘和实盘交易链路打通

- 风险规则可配置并有效拦截

### 11.3 阶段三：治理可用

- 审批、审计、告警和运维可用
- 多角色协作和权限隔离可用
- 生产运行可追踪、可恢复

## 12. 结论

这套量化平台的核心不是单一策略能力，而是把数据、研究、组合、交易、风险、治理整合成一个长期可演进的平台。需求设计必须完整覆盖“研发效率、交易安全、结果可解释、过程可追溯、系统可治理”五个维度。

如果只做一个功能集合，它会很快演变成脚本仓库；只有把需求定义成平台能力，后续研发和运营才能真正形成复利。
