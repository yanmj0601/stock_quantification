# 当前系统差距矩阵与后台管理界面设计

## 1. 文档目标

本文档用于基于当前 `stock_quantification` 项目实际实现情况，梳理：

1. 当前系统已经具备哪些能力
2. 与目标量化平台相比还缺哪些关键能力
3. 各能力缺口的优先级
4. 为补齐这些能力应建设哪些后台管理界面

本文档强调“现状对目标”的差距，不重复展开完整系统设计正文。

关联文档：

- `docs/quant_framework_system_design.md`
- `docs/quant_requirements_design.md`
- `docs/quant_implementation_spec.md`

---

## 2. 当前系统总体判断

### 2.1 当前系统处于什么阶段

当前系统更接近：

- 一个已经具备研究、回测、仿真执行能力的量化底座
- 一个适合单人或小团队内部试验的原型平台
- 一个“研究闭环基本成立，但平台治理和交易闭环尚未完整”的系统

当前系统还不完全是：

- 正式的数据平台
- 可多角色协作的量化中台
- 可稳定生产运行的交易平台
- 完整的后台管理系统

### 2.2 当前系统的优势

- 研究主链路已经成型
- A 股 / 美股统一语义已经具备
- 回测、验证、paper 语义已经打通
- 已有基础 Web 工作台
- 已有文件系统结果索引和研究结果中心雏形
- 已有最小多 Agent 编排

### 2.3 当前系统的主要短板

- 数据治理不足
- 状态持久化仍偏轻量
- 交易闭环仍未覆盖 broker 回报与对账
- 策略治理不足
- 因子治理不足
- 平台治理不足
- 后台管理视图仍以只读工作台为主

## 3. 差距矩阵

说明：

- `已有`：当前已具备且基本可用
- `部分具备`：已有雏形，但无法支撑平台化
- `缺失`：当前基本未建设

## 3.1 总体差距矩阵

| 模块域 | 当前状态 | 现状判断 | 主要缺口 | 优先级 |
|---|---|---|---|---|
| 统一领域模型 | 已有 | 基础较好 | 需要扩展版本化与治理对象 | P1 |
| 数据抓取与缓存 | 部分具备 | 研究可用 | 缺数据仓库、数据源管理、质量治理 | P0 |
| 证券主数据与市场规则 | 部分具备 | 能支撑当前策略 | 缺正式主数据中心、规则版本治理 | P0 |
| 因子计算 | 部分具备 | 已能计算核心因子 | 缺因子注册、版本、评分卡、淘汰机制 | P0 |
| 研究项目与实验管理 | 缺失 | 只有脚本和 artifact | 缺研究项目、实验台、场景管理 | P0 |
| 策略管理 | 部分具备 | 有策略预设 | 缺策略模板、参数集、版本状态流转 | P0 |
| 回测与验证 | 部分具备 | 能完成基础研究验证，且已有结果索引与统一摘要 | 缺统一任务管理、结果中心筛选/对比、批量实验视图 | P0 |
| 组合管理 | 部分具备 | 已有规则式组合生成 | 缺优化器接口、约束中心、容量分析 | P1 |
| 风控管理 | 部分具备 | 有风险校验 | 缺研究/组合/交易/运行四层风控中心 | P0 |
| 订单与执行 | 部分具备 | 已有建议链路、最小执行和本地 paper 账本闭环 | 缺订单状态机、成交回写、撤改单、对账 | P0 |
| 账户管理 | 部分具备 | 有本地 paper 和账户状态 | 缺正式账户中心、多账户治理、资金分配 | P1 |
| 券商接入 | 部分具备 | 仅有 Alpaca paper 基础能力 | 缺回报同步、异常处理、适配器体系 | P1 |
| 任务调度与运维 | 部分具备 | 有 ops 页面、任务锁和审计/任务日志展示 | 缺调度中心、异步任务、告警体系 | P0 |
| 权限与审批 | 缺失 | 基本未建设 | 缺用户角色、审批流、发布治理 | P1 |
| API 平台 | 缺失 | 当前偏 CLI/Web 原型 | 缺标准 API 和开放接口 | P1 |
| 报表中心 | 部分具备 | 有 artifact、结果索引和局部统一摘要展示 | 缺统一报表中心、导出中心、归因中心 | P1 |

## 6. 后台管理系统设计总览

## 6.1 设计目标

后台管理系统不只是“给现有脚本加个页面”，而是要承担以下职责：

- 把平台对象管理化
- 把任务运行可视化
- 把风险和异常收敛到统一工作台
- 把策略与数据治理流程标准化
- 把上线和生产运行变成可审批、可追溯流程

## 6.2 信息架构建议

建议后台采用以下一级导航：

1. 总览看板
2. 数据中心
3. 研究中心
4. 策略中心
5. 回测验证
6. 交易中心
7. 风险中心
8. 账户中心
9. 运维中心
10. 治理中心

## 6.3 角色视角

### 量化研究员常用

- 数据中心
- 研究中心
- 策略中心
- 回测验证

### 投资经理常用

- 总览看板
- 策略中心
- 回测验证
- 账户中心
- 风险中心

### 交易运营常用

- 交易中心
- 账户中心
- 风险中心
- 运维中心

### 风控负责人常用

- 风险中心

### 平台运维常用

## 9. 推荐建设顺序

### 第一阶段

- 数据中心
- 研究中心

理由：

- 支持团队协作和生产治理

## 10. 结论

当前系统最需要完善的，不是单点策略能力，而是把已有研究底座升级为平台对象、任务对象、交易对象和治理对象都可管理、可配置、可追溯的后台系统。
